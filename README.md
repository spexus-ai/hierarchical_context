# Context Tree Repository

Репозиторий-шаблон для древовидной архитектуры распространения информации.

Цель структуры:
- давать короткий релевантный summary для принятия решения;
- обеспечивать drill-down до первоисточников;
- удерживать общую картину мира в актуальном состоянии;
- разделять глобальный, доменный, командный и операционный контекст.

## Уровни

- `L0-global-picture` — глобальная картина продукта и компании.
- `L1-domains-streams` — домены, value streams и крупные зоны продукта.
- `L2-capabilities-teams` — зоны ответственности команд и подсистем.
- `L3-initiatives-changes-decisions` — инициативы, изменения, решения, инциденты.
- `L4-evidence-sources` — первичка: PR, ADR, метрики, тикеты, обсуждения, отчёты.

## Базовый принцип

Навигация должна идти сверху вниз:

`L0 -> L1 -> L2 -> L3 -> L4`

При этом на практике связи не обязаны быть строго древовидными. Поэтому:
- дерево используется как основной способ навигации;
- кросс-ссылки между уровнями обязательны;
- summary каждого узла должен ссылаться вниз на более детальные артефакты.

## Правила ведения

1. Каждый каталог уровня обязан содержать `README.md`.
2. Каждый `README.md` отвечает на два вопроса:
   - что это за уровень;
   - какие решения на нём принимаются.
3. Каждый уровень должен явно ссылаться:
   - вверх: на контекст-родитель;
   - вниз: на дочерние узлы / артефакты.
4. Для всех summary указывать:
   - `updated_at`
   - `owner`
   - `status`
   - `links`
5. Источник истины по фактам — всегда уровень `L4`.

## Рекомендуемые сценарии использования

- ежедневная синхронизация продуктовых и инженерных команд;
- подготовка релизов;
- анализ системных рисков;
- разбор инцидентов;
- архитектурные изменения;
- онбординг новых участников;
- работа ИИ-агентов с многоуровневым контекстом.
