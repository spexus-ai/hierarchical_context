# Context Tree Repository

Репозиторий-шаблон для древовидной архитектуры распространения информации.

Цель структуры:
- давать короткий релевантный summary для принятия решения;
- обеспечивать drill-down до первоисточников;
- удерживать общую картину мира в актуальном состоянии;
- разделять глобальный, доменный, командный и операционный контекст.
- поддерживать мультикомандный и мультипроектный портфель в общем knowledge git repository.

## Уровни

- `L0-global-picture` — глобальная картина продукта и компании.
- `L1-domains-streams` — домены, value streams и крупные зоны продукта.
- `L2-capabilities-teams` — зоны ответственности команд и подсистем.
- `L3-initiatives-changes-decisions` — инициативы, изменения, решения, инциденты.
- `L4-evidence-sources` — первичка: PR, ADR, метрики, тикеты, обсуждения, отчёты.

## Базовый принцип

Навигация должна идти сверху вниз:

`L0 -> L1 -> L2 -> L3 -> L4`

При этом на практике связи не обязаны быть строго древовидными. Поэтому:
- дерево используется как основной способ навигации;
- кросс-ссылки между уровнями обязательны;
- summary каждого узла должен ссылаться вниз на более детальные артефакты.

## Portfolio / Multi-project model

Структура рассчитана на отдельный knowledge git repository, в который могут писать несколько команд и агентов.

Основные правила:
- прозрачность: любой проект портфеля виден всем;
- кросс-перенос знаний: evidence и change streams должны явно отмечать, где знание полезно другим проектам;
- мультиагентность / мультикомандность / мультипроектность: разные команды обновляют свои зоны, но работают в общей модели стратегии;
- одна стратегическая цель L0 может зависеть от нескольких проектов;
- один проект может вносить вклад в несколько стратегических целей.

Рекомендуемая структура L4:
- `portfolio/<portfolio-id>/projects/<project-id>/entries/YYYY/MM/DD/<entry>.md`
- `portfolio/<portfolio-id>/projects/<project-id>/journal.md`
- `shared-intelligence/<stream-id>/entries/...`
- `triage/inbox/...`

Правило хранения:
- один проект = одна папка;
- одна запись evidence = один файл;
- `journal.md` проекта — это индекс и хронологическая лента, но не источник истины;
- canonical evidence не дублируется между проектами, связи оформляются через cross-links и metadata.

## Правила ведения

1. Каждый каталог уровня обязан содержать `README.md`.
2. Каждый `README.md` отвечает на два вопроса:
   - что это за уровень;
   - какие решения на нём принимаются.
3. Каждый уровень должен явно ссылаться:
   - вверх: на контекст-родитель;
   - вниз: на дочерние узлы / артефакты.
4. Для всех summary указывать:
   - `updated_at`
   - `owner`
   - `status`
   - `links`
5. Источник истины по фактам — всегда уровень `L4`.
6. Каждый `L3`-узел обязан принадлежать конкретной зоне ответственности `L2`.
7. Каждая `L2`-зона должна быть изолирована по ownership boundary: репозиторий, сервис, capability, аналитический контур или другой отдельный asset.
8. Ежедневный проход сначала публикует evidence в git, потом поднимает агрегацию вверх по дереву.

## Рекомендуемые сценарии использования

- ежедневная синхронизация продуктовых и инженерных команд;
- подготовка релизов;
- анализ системных рисков;
- разбор инцидентов;
- архитектурные изменения;
- онбординг новых участников;
- работа ИИ-агентов с многоуровневым контекстом.
